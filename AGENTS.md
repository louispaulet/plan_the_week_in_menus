# AGENTS.md — Week Menu Planner

## Project Mission

Build a local-first-feeling weekly menu planning application that helps users plan 2 meals per day across a full week while respecting dietary, nutritional, budget, preference, and household rules.

The product uses `gpt-5-nano` through backend-only OpenAI API calls. The user should never experience the app as a chat interface. The LLM must not stream raw prose into the UI. Instead, the LLM acts as a planning engine that returns structured JSON used to generate reusable interfaces: buttons, cards, forms, warnings, meal suggestions, and week schedule updates.

The user experience should feel like a smart interactive planning tool, not like talking to a bot.

## Tech Stack

### Frontend

- Vite
- React
- TailwindCSS
- Plain client-side routing unless a router becomes necessary
- Minimal dependencies unless there is a strong product reason

### Backend

- Cloudflare Worker
- Singular JavaScript file as the main backend entry point
- Backend owns all OpenAI API calls
- Backend validates all LLM output before returning it to the frontend

### Database

Use a Cloudflare-compatible database approach for persistence.

Preferred options:

1. Cloudflare D1 for relational structured data
2. Cloudflare KV for simple reusable blocks, settings, cached interface templates, and prompt-context snippets
3. Cloudflare R2 only if later we need larger assets or exports

Initial implementation should favor D1 for core entities and optionally KV for reusable prompt/interface blocks.

## Core Product Principles

### 1. No Raw LLM Chat

Do not build a chat UI.

The LLM returns structured JSON only. The frontend renders the JSON as deterministic UI components.

Good:

- Buttons with 3–4 choices
- Ingredient chips
- Editable rule blocks
- Warning cards
- Generated forms
- Meal suggestion cards
- Schedule cells
- Confirmation modals

Bad:

- Chat bubbles
- Raw assistant paragraphs
- Streaming wall-of-text responses
- “Ask me anything” input as the main interface

### 2. Interface-First AI

The LLM should generate interface specifications, not prose.

Every AI response should map to one or more of these UI primitives:

- `choice_buttons`
- `meal_cards`
- `rule_warning_cards`
- `form_fields`
- `ingredient_chips`
- `schedule_patch`
- `preference_block`
- `shopping_list_patch`
- `explanation_tooltip`
- `next_step_panel`

### 3. Reusable Interface Catalog

Interfaces can be either:

1. Generated dynamically by the LLM
2. Loaded from a catalog of known reusable interface templates

The catalog should include reusable components like:

- Single text input + submit button
- Multi-select ingredient chips
- 3-option meal choice panel
- Rule editor block
- Weekly schedule grid
- Meal detail editor
- Warning card stack
- Save/load rule set dialog
- Drag-and-drop reusable block picker

Prefer loading from the catalog when possible. Generate only when the current state requires a custom layout.

### 4. Rules Are First-Class Objects

Rules are not just prompt text. They are saved, reusable, versioned planning constraints.

Examples:

- “Low carb during weekday dinners.”
- “At least 5 fruits and vegetables per day.”
- “Avoid bell peppers.”
- “Prefer air fryer compatible dishes.”
- “Budget: around €X per week.”
- “Use leftovers when possible.”

Each rule should support:

- Display name
- Full text
- Severity: `hard`, `soft`, `warning`
- Scope: `global`, `day`, `meal`, `ingredient`, `dish`, `shopping_list`
- Status: `active`, `inactive`, `archived`
- Optional tags
- Optional structured extraction generated by the backend

### 5. Back Blocks / Context Injection

A saved rule block should be usable without the user needing to reread or rewrite the full text.

The product should support “back blocks”: saved reusable context blocks injected into backend prompts.

Example flow:

1. User writes or imports a rule.
2. The backend normalizes it into a reusable rule block.
3. User saves the block.
4. Later, user drags the block into a plan.
5. Backend injects the block into the LLM planning context.
6. Frontend shows only the compact rule label unless the user expands it.

The app should make reusable planning context feel physical and modular.

### 6. Frequent 3–4 Option Decisions

The product should avoid open-ended paralysis.

Most generated steps should offer 3–4 clickable options.

Examples:

- “Pick a planning style”
  - Healthy and cheap
  - Low carb
  - Fast weekday meals
  - Use what I already have

- “Choose dinner direction for Tuesday”
  - Chicken salad bowl
  - Air fryer salmon and vegetables
  - Mushroom omelette
  - Suggest 3 new options

- “Rice conflicts with low carb. What should we do?”
  - Use a very small portion
  - Replace with cauliflower rice
  - Move it to lunch
  - Ignore warning once

### 7. Rule-Breaking Detection

The app must highlight conflicts between the week plan and active rules.

Examples:

- Low carb rule + rice dish → warning
- 5 fruit/veg/day rule + day has only 2 servings → warning
- Avoid ingredient rule + selected dish contains ingredient → hard conflict
- Budget rule + expensive dish combination → warning

Warnings should be actionable, with buttons:

- Replace dish
- Reduce portion
- Accept exception
- Show alternatives
- Edit rule

### 8. Week Schedule Visualization

The main view should clearly show:

- Monday to Sunday
- Lunch and dinner for each day
- Meal cards inside each cell
- Rule status per day
- Fruit/vegetable count per day
- Carb/fat/protein rough indicators if available
- Warnings as small badges
- Empty slots as clear calls to action

A good default layout:

- Desktop: 7-column grid, each day as a column, lunch/dinner stacked
- Mobile: day-by-day vertical cards

## Agent Roles

Codex should implement the app through these conceptual agents. These are not necessarily runtime agents; they are product and code responsibilities.

### Product Architect Agent

Responsibilities:

- Preserve the no-chat UX principle
- Keep user flows button-driven and interface-driven
- Ensure all AI output becomes structured UI state
- Prevent feature creep
- Keep the week planner central

### Frontend Agent

Responsibilities:

- Build clean Vite + React + TailwindCSS components
- Render backend interface specs safely
- Implement schedule visualization
- Implement rule block editor and reusable block library
- Implement drag-and-drop or simple click-to-add behavior for saved rules
- Keep state management simple and explicit

Frontend must not call OpenAI directly.

### Backend Agent

Responsibilities:

- Implement Cloudflare Worker endpoints
- Store and retrieve preferences, rules, meal plans, reusable blocks, and interface templates
- Call OpenAI with `gpt-5-nano`
- Validate LLM JSON output
- Convert failures into safe fallback UI specs
- Never expose API keys to frontend

### Database Agent

Responsibilities:

- Design schema for users, preferences, rules, reusable blocks, meal plans, meals, warnings, and interface templates
- Provide migrations or schema comments
- Keep persistence simple for the MVP
- Avoid over-normalization unless needed

### Prompt Engineering Agent

Responsibilities:

- Design prompts that request strict JSON only
- Include active rules and week state as compact context
- Ask for 3–4 options when appropriate
- Require warnings when rules are broken
- Require UI specs instead of prose
- Keep prompts cheap and suitable for `gpt-5-nano`

### QA Agent

Responsibilities:

- Test that no raw LLM text leaks into the UI
- Test invalid JSON fallback behavior
- Test rule conflict detection
- Test saving/loading rules
- Test week schedule rendering
- Test mobile and desktop layouts

## LLM Contract

All backend LLM calls should request JSON matching a known schema.

Example top-level shape:

```json
{
  "type": "planner_response",
  "version": 1,
  "summary": {
    "machine_label": "low_carb_week_planning_step",
    "user_visible_title": "Choose your next dinner option"
  },
  "ui": [
    {
      "component": "choice_buttons",
      "props": {
        "question": "Tuesday dinner needs a low-carb option. Pick one:",
        "choices": [
          {
            "label": "Air fryer salmon + green beans",
            "action": {
              "type": "select_meal_option",
              "payload": {
                "day": "tuesday",
                "slot": "dinner",
                "meal_id": "generated_salmon_green_beans"
              }
            }
          }
        ]
      }
    }
  ],
  "warnings": [],
  "schedule_patch": null,
  "saved_blocks_suggestions": []
}
```

The frontend should render only known `component` values. Unknown components should be ignored or rendered as safe fallback cards.

## Implementation Rules

1. Never call OpenAI from the frontend.
2. Never trust LLM output without validation.
3. Never render arbitrary HTML from the LLM.
4. Render only whitelisted UI components.
5. Prefer structured actions over free text.
6. Make every warning actionable.
7. Keep the week grid visible as the product anchor.
8. Keep reusable rules and blocks easy to save and reload.
9. Default to 3–4 button choices.
10. Treat `gpt-5-nano` as a cheap planning copilot, not as the UI itself.

## MVP Scope

Build these first:

1. Week grid with 7 days and 2 meals per day
2. Rule editor and rule list
3. Save/load rules
4. Create a new weekly plan
5. Suggest 3 meal options for an empty slot
6. Insert chosen meal into the grid
7. Check active rules and show warnings
8. Backend OpenAI calls through Worker only
9. D1 schema for persistence
10. Interface renderer for known UI primitives

## Definition of Done

The app is successful when a user can:

1. Define reusable dietary and preference rules
2. Start a weekly meal plan
3. Fill lunch and dinner slots with guided button-driven suggestions
4. See when the plan breaks rules
5. Fix conflicts through suggested actions
6. Save and reload both plans and rule blocks
7. Complete the process without ever reading raw LLM prose
